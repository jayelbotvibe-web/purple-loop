Purple Loop System Architecture & Evidence — v1.2.0

Risk-driven detection validation. Emulates prioritized ATT&CK techniques in an isolated Wazuh lab and proves whether your Sigma detections catch them — with evidence.

1. The Intel-to-Detection Pipeline

🔴 threat-intel-arbiter

Ingests CISA KEV + MISP feeds. Scores threats with SSVC. Outputs a priority-ordered list of CVEs mapped to ATT&CK techniques.

1,147 alerts → top 10 export

→

🟣 Purple Loop

Loads the prioritized plan. Executes atomics on lab victims. Collects real Wazuh telemetry. Evaluates Sigma rules. Produces an evidence-backed coverage report.

CVE → technique → atomic → verdict

3. Verdict Semantics — What Each Result Means

DETECTED
Sigma rule condition matched ≥1 event. Rule file named in rule_matched. Matching events in evidence[].
PARTIAL
Primary selection matched, required secondary did not (e.g. command seen, parent context missing).
MISSED
Telemetry was collected, but no mapped Sigma rule matched. This is a genuine detection gap — the valuable finding.
NO_TELEMETRY
Zero events in the collection window. A collection problem, not a detection result. Never counted as coverage.
INCONCLUSIVE
Pipeline canary failed. Run is not trusted. Coverage is not reported. Non-zero exit.

4. Pipeline Canary — The Positive Control

Before any real techniques run, Purple Loop fires a pipeline canary — a known-benign command engineered to be trivially detectable. It proves the full pipeline (execute → telemetry → collect → normalize → match) is healthy per platform.

✔
Canary DETECTED
Pipeline healthy.
Every MISSED is a genuine gap.
✘
Canary NOT detected
Run is INCONCLUSIVE.
Coverage not reported.
Pipeline is broken.
$ make canary
